- **Decision:** Считать настройки, историю терминалов, историю AI-чатов, session metadata, host/tunnel profiles и workspace state чувствительными данными и хранить их в зашифрованном виде по умолчанию.
- **Reason:** Пользовательский сценарий включает удалённые машины, AI workflows и рабочую историю; утечка таких данных разрушает модель доверия системы.
- **Context:** В Rust-core документах добавлены явные требования `encrypted by default`, history retention/erase semantics, запрет plaintext persistence в production mode и отдельный `history-store`.

## [2026-04-08] Storage Decision For Rust Core
- **Decision:** Выбрать `encrypted embedded database` как canonical operational store и `encrypted portable files/bundles` как формат импорта, экспорта, переноса и backup.
- **Reason:** Для ядра нужны транзакционность, миграции, связанное состояние между sessions/hosts/tunnels/history/workspaces и быстрые запросы; это делает БД сильнее чисто файловой модели. При этом переносимость и быстрый deploy требуют файлового exchange layer.
- **Context:** В [RUST_CORE_PLAN.md](/home/mrz/projects/edex-ui-2026/RUST_CORE_PLAN.md) закреплена модель `DB as source of truth, files as exchange format`, с практическим ориентиром на SQLite/SQLCipher-compatible path и encrypted export bundles.

## [2026-04-08] AI-Optimized History And File Access
- **Decision:** История, файлы и workspace state должны проектироваться не только как encrypted persistence, но и как `AI-addressable context plane` с быстрыми daemon-owned retrieval APIs.
- **Reason:** Глубокая интеграция AI/agents требует не медленного повторного обхода файловой системы и логов, а сверхбыстрого локального доступа к релевантной истории, recent file changes, transcript chunks и workspace state. Иначе агентный слой станет дорогим, медленным и ненадёжным.
- **Context:** В Rust-core документах добавлены требования к `structured chunking`, `history-store`, `file-index` и новому `context-engine`; canonical encrypted DB остаётся source of truth, поверх неё строятся encrypted derived indexes/read models для low-latency agent retrieval.

## [2026-04-08] MemPalace Reference
- **Decision:** Использовать `milla-jovovich/mempalace` как референс для local-first AI memory subsystem, но не как прямой шаблон хранилища всего ядра.
- **Reason:** Проект хорошо показывает ценность raw-verbatim storage, local semantic retrieval, MCP tooling и отдельного knowledge-graph слоя, но его текущая архитектура ориентирована именно на memory/RAG system, а не на полный session/remote/workspace core.
- **Context:** По README и package metadata `mempalace` хранит verbatim exchanges в ChromaDB, использует semantic search, отдельный SQLite-based temporal knowledge graph, conversation/file miners, MCP server с 19 tools и local-only execution. При этом maintainers сами публично признали ограничения и исправляют overstated claims, wiring gaps и security issue в hooks. Для нашего `rust-core` полезны идеи: raw history as asset, normalized ingest, derived retrieval views, MCP-compatible tool surface, separate factual graph; не полезно слепо копировать ChromaDB-first storage как canonical state layer для всего продукта.

## [2026-04-08] Linux Key Storage Strategy
- **Decision:** Хранить master encryption key через strongest practical Linux mechanism с явной иерархией fallback, не привязывая весь runtime к `systemd-creds`.
- **Reason:** Требование "всё шифруется" бессмысленно без корректной стратегии хранения ключей; безопасность должна обеспечиваться не только cipher-ом, но и key lifecycle.
- **Context:** В актуальном состоянии план исходит из user-session friendly порядка: `Secret Service/system keyring` для стандартных Linux deployments, затем deployment-specific `systemd-creds`/TPM2 integration при явной поддержке окружения, затем passphrase fallback для portable/headless режимов. Private keys, terminal history, AI chat history и settings store не должны сохраняться в plaintext в production mode.

- **Reason:** Повторный WAR ROOM показал, что функционально ядро уже живо, но AI-native trust model ещё была слишком мягкой: `danger-full-access` можно было просить напрямую, file API работал по всем известным путям без жёсткой workspace границы, а agent `cwd` можно было увести через lexical `..` path traversal.
- **Context:** В [core-policy/src/lib.rs](/home/mrz/projects/edex-ui-2026/rust/crates/core-policy/src/lib.rs) добавлены `enforce_agent_policy()`, quotas (`DEFAULT_MAX_IN_MEMORY_HISTORY_ENTRIES`, `DEFAULT_MAX_INDEXED_ROOTS`, `DEFAULT_MAX_INDEXED_ENTRIES_PER_ROOT`, preview caps) и `ensure_path_within_roots()`; в [runtime-daemon/src/lib.rs](/home/mrz/projects/edex-ui-2026/rust/crates/runtime-daemon/src/lib.rs) `RunAgentTask` теперь пропускается через policy decision point, `RegisterWorkspace` канонизирует roots, `build_agent_task()` канонизирует `cwd` и режет `root/../escape`, history держится bounded in-memory, file surfaces (`RefreshFileIndex`, `FileList`, `FileStat`, `FilePreview`, `FileSearch`) ограничены workspace roots и quotas; в [core-state/src/lib.rs](/home/mrz/projects/edex-ui-2026/rust/crates/core-state/src/lib.rs) добавлен `StateError::InvalidWorkspaceRoot`; в [file-index/src/lib.rs](/home/mrz/projects/edex-ui-2026/rust/crates/file-index/src/lib.rs) добавлены `AccessDenied` и `EntryLimitExceeded`; в [runtime-daemon/src/bin/host_acceptance.rs](/home/mrz/projects/edex-ui-2026/rust/crates/runtime-daemon/src/bin/host_acceptance.rs) acceptance harness синхронизирован с roots-aware contract. Подтверждение: `cargo check --workspace --all-targets`, `cargo test --workspace`, `cargo clippy --workspace --all-targets -- -D warnings` зелёные; добавлены доказательные тесты `agent_policy_denies_danger_full_access_roundtrip`, `agent_task_rejects_dotdot_escape_outside_workspace`, `in_memory_history_retention_is_bounded`; host-side `./target/debug/host_acceptance` снова завершился `HOST ACCEPTANCE OK`. Остаточный риск зафиксирован честно: recovery export/import/rekey всё ещё материализует bundle целиком в память, то есть это peak-memory risk на очень больших архивах, но не классическая утечка.
